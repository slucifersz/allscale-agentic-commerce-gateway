The real problem

The agent can pay. The question is whether the merchant should accept it.

"Can it pay?" is solved. "Should we accept it?" is not.

Dirty funds

The payer's funds may trace back to hacks, stolen wallets, mixers, or sanctioned addresses. The transfer succeeds either way — the merchant only finds out later.

EXAMPLE
500 USDC for a license; payer address recently interacted with a flagged high-risk address → paid, but high-risk

Unauthorized agent

An agent making a payment is not the same as its principal authorizing that payment. Nothing in the transfer itself proves the mandate covers it.

EXAMPLE
mandate covers $20; agent paid $200 → payment exceeds mandate

Black-box payment

The merchant sees an address, a request, and a tx hash — not why the payment was made or which order it belongs to.

EXAMPLE
requirement: order context · today provides: an address and a hash

High-value / resellable

Cloud credits, gift cards, annual SaaS — once delivered, they are hard to claw back if the funds turn out to be tainted.

EXAMPLE
1000 USDC cloud credits delivered, funds later flagged → no recourse

Regulated

Financial, cross-border, and enterprise SaaS merchants must know who the customer is, screen against sanctions lists, and keep an audit trail.

EXAMPLE
requirement: KYC'd counterparty · sanctions screen · audit log
agent payment today provides: none of the three

Low-risk is fine

Small, low-risk payments are genuinely fine to accept as-is. The verification layer earns its keep on high-value, resellable, regulated, enterprise, and cross-border payments.

EXAMPLE
small, low-risk → accept as-is
Verification layer — where AllScale adds the moat

The router accepts the payment. The verification layer decides if it's safe to settle.

GATE 1

Fund-source screening

Screens where the money came from — before it settles.

Built on AllScale's built-in KYT (BlockSec). Every incoming payment's source address is checked against high-risk signals — mixers, sanctioned addresses, addresses with recent exposure to stolen or laundered funds.

Checks: source reputation · sanctions exposure · blocklist match

KYT · MOAT
GATE 2

Payer verification: identity + mandate

Proves the payer is authorized — without revealing who they are.

The paying agent presents two Primus zkTLS attestations: one proving its principal passed KYC at a regulated institution, another proving the principal's own system of record holds a live mandate authorizing this exact payment. AllScale verifies both; the attested data lives in systems AllScale does not control.

Checks: principal KYC attestation · live payment mandate · selective disclosure

AUTHROADMAP

Primus attests that the data is real; the regulated institution and the principal's own systems produce the KYC and authorization states. AllScale is the verifier, not the data source.

Spending-limit judgment runs on an on-chain contract, not through Primus. Mandate cap (per-payment authorization, off-chain attestation) and spending limit (cumulative cap, on-chain contract) are two separate controls.

Passed → settle; either gate fails → blocked before settlement.

A GENERIC GATEWAY

only answers "Did the money arrive?"

ALLSCALE

answers "Money arrived; source passed KYT; a live mandate from the principal covers this payment; order released with verified payment evidence."

Settlement model

How a verified payment settles on HashKey

V1 is intentionally narrow — stablecoins on HashKey first, no card rails, no custodial merchant balances.

HashKey Chain configuration

  • Mainnet chain ID: 177 — LIVE
  • Native token: HSK
  • Mainnet stablecoin: USDC.e (bridged USDC)
  • Settlement contract: SettlementGateway
  • Testnet (chain ID 133): mUSDC · mUSDT test tokens

Settlement path

1
Gateway signs checkout terms

An EIP-712 request fixes checkout ID, token, amount, treasury, expiry, and metadata hash.

2
Payer submits the router transaction

The contract moves exact USDC/USDT from payer to merchant treasury via transferFrom.

3
On-chain guards run

Per-agent cumulative spending limit, signature replay protection, and checkout expiry are enforced in the contract.

4
Gateway verifies the receipt

The order is committed only after matching the on-chain CheckoutSettled event back to the canonical checkout.

Why AllScale

Routing is table stakes. Compliance is the moat.

Many teams can route agent protocols. What they don't have is the compliance layer. AllScale already runs built-in KYT and automated KYC/KYB, plus self-custody stablecoin settlement. Combining a multi-protocol router with that existing compliance foundation is what makes agent payments actually safe for merchants to accept — and it's why this is AllScale's to build.

We don't build the agent. The agent is the payer we serve — a trend we position for, not a product we make.
